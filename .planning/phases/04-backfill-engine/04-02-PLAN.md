  <name>Task 1: BackfillEngineImpl with pgx.Conn callback and AppendAndQueue integration</name>
  <files>internal/backfill/backfill.go</files>
  <action>
    Implement `BackfillEngineImpl` (the concrete struct satisfying `BackfillEngine`) in `internal/backfill/backfill.go`. The struct receives its dependencies at construction time and avoids opening its own database connections internally.

    Constructor:
    ```go
    // AppendFn is the function BackfillEngineImpl calls to deliver each snapshot event.
    // In production this is connector.AppendAndQueue. In tests it can be a mock.
    type AppendFn func(ctx context.Context, ev *event.ChangeEvent) error

    // OpenConnFn opens a pgx.Conn for snapshot SELECT queries.
    // The backfill engine is NOT allowed to use the replication connection.
    type OpenConnFn func(ctx context.Context) (*pgx.Conn, error)

    type BackfillEngineImpl struct {
        configs    []BackfillConfig
        store      BackfillStore
        idGen      *event.IDGenerator
        appendFn   AppendFn
        openConnFn OpenConnFn
    }

    func NewBackfillEngine(
        configs    []BackfillConfig,
        store      BackfillStore,
        idGen      *event.IDGenerator,
        appendFn   AppendFn,
        openConnFn OpenConnFn,
    ) *BackfillEngineImpl
    ```

    `HasPendingBackfills()`: For each config, load state. If strategy is "stream_only" or state.Status == "completed" or "deferred", skip. Return true if any table has status "pending" or "running". If store returns error, log it and return false.

    `Run(ctx context.Context) error`: For each config in order:
    1. Load state. If nil (first run), initialize state with Status="pending", Strategy=config.Strategy.
    2. If strategy == "stream_only": save state with Status="completed". Continue.
    3. If strategy == "snapshot_deferred": save state with Status="deferred". Continue.
    4. If state.Status == "completed": skip.
    5. Otherwise: call `c.snapshotTable(ctx, config, state)`.
    6. On snapshotTable success: save state with Status="completed". Emit snapshot_complete control event via appendFn.

    `snapshotTable(ctx, config, state)`:
    1. Open pgx.Conn via openConnFn. Defer close.
    2. Estimate total rows: `SELECT GREATEST(reltuples::bigint, 0) FROM pg_class WHERE relname = $1`. If <= 0, set TotalRows=-1 (unknown). Otherwise update state.TotalRows.
    3. Initialize KeysetCursor from config.PKCols. If state.CursorKey is non-nil, unmarshal it into cursor.LastPK.
    4. Initialize BatchOptimizer.
    5. Loop:
       a. Build query via cursor.buildFirstQuery or buildNextQuery based on whether cursor.LastPK is nil.
       b. Record batchStart := time.Now().
       c. Execute query on pgx.Conn (not the replication connection — this is the separate query conn).
       d. For each row: scan into []any via rows.Values(). Marshal to rowJSON. Extract PKValues from PK columns. Marshal PKValues to pkJSON. Call WatermarkChecker (if non-nil) ShouldEmit — if false, skip row. Otherwise call MakeReadEvent then appendFn.
       e. Update cursor.LastPK with last row's PK values. Marshal to state.CursorKey. Increment state.ProcessedRows by batch count.
       f. Save state via store.SaveState (BKF-03 — crash-resumable).
       g. Call optimizer.Adjust(time.Since(batchStart)) for next batch size.
       h. If batch returned < batchSize rows, snapshot is complete — break loop.
    6. Return nil.

    WatermarkChecker is an optional field. Pass nil in stream_only/snapshot_only tests. In production, the connector constructs it with the live EventLog.

    Pitfall: Rows.Values() returns []any with pgx-decoded Go native types (int64 for bigint, string for text, etc.). Use `pgx/v5` — the import is already in go.mod. Do not use database/sql Scan with explicit dest pointers; use pgx rows.Values() for dynamic column scanning.

    Pitfall: Do NOT use the replication replConn for snapshot queries. The openConnFn callback is what provides the separate query connection. The connector will pass `pgx.Connect(ctx, c.cfg.DSN)` as the openConnFn.

    Add constructor:
    ```go
    func NewWithBackfill(
        cfg    Config,
        store  checkpoint.CheckpointStore,
        idGen  *event.IDGenerator,
        el     eventlog.EventLog,
        bf     backfill.BackfillEngine,
    ) *PostgresConnector {

    ```

    In `connectAndStream`, after `pglogrepl.StartReplication` succeeds and before calling `receiveLoop`, add:

    ```go
    // Launch backfill goroutine if backfill engine has pending work.
    // The backfill engine delivers events via AppendAndQueue (same path as WAL events),
    // which serializes Append calls — no concurrent Badger writes (Pitfall 2 in research).
    if c.backfillEng != nil && c.backfillEng.HasPendingBackfills() {
        go func() {
            if err := c.backfillEng.Run(ctx); err != nil && ctx.Err() == nil {
                slog.Error("backfill engine: run failed", "error", err)
            }
        }()
    }
    ```

    Important: The backfill goroutine calls AppendAndQueue, which already holds the serialization contract (events channel). No additional mutex is needed because AppendAndQueue sends to the buffered events channel and the eventLog.Append call happens inside AppendAndQueue on the caller's goroutine. Concurrent Append calls from two goroutines IS a risk — review AppendAndQueue implementation. If Append is called directly (not via channel), wrap with sync.Mutex. Per the research doc (Pitfall 2), the recommended pattern is a single dispatcher goroutine, but if AppendAndQueue already serializes via a channel then it is safe.

    Verify the existing AppendAndQueue implementation: if it calls eventLog.Append directly (not via a channel), add a `sync.Mutex appendMu` field to PostgresConnector and lock it around the Append call inside AppendAndQueue. Do not restructure the method — just add the mutex guard.

    The connector's `cfg.DSN` is available for the openConnFn. The backfill engine receives this via its constructor — the connector does NOT call pgx.Connect itself. The caller (cmd/kaptanto wiring) constructs BackfillEngineImpl with the openConnFn closure.
