## Tech Debt by Phase

### Phase 20: SQS Sink

**SQSSinkConfig.TLS inert for custom CA** (CFG-03 partial)
- File: `internal/output/sqs/consumer.go` — `NewSQSSinkConsumer` never reads `cfg.TLS`
- Impact: Users who configure a custom CA file for SQS VPC endpoint pinning will have the config silently ignored. AWS SDK enforces HTTPS for all SQS calls by default, so transport TLS is always on — only custom CA pinning is missing.
- Severity: Low — affects advanced VPC endpoint configurations only

### Phase 22: Google Pub/Sub Sink

**PubSubSinkConfig.TopicTemplate not applied per-message** (CFG-02 partial for PubSub)
- File: `internal/output/pubsub/consumer.go` — `topicT` field allocated but `Deliver` always publishes to `cfg.TopicID`
- Explicit code comment: "TopicTemplate is preserved in the config for future multi-topic support but is not applied per-message in v2.6.0."
- Impact: Users who configure `topic-template` for per-table routing in the PubSub sink will not get per-table routing — all events go to the single `topic-id`. Per-key ordering works correctly via `OrderingKey`.
- Severity: Medium — silent functional limitation for multi-topic PubSub configurations

### Cross-Phase (root.go)

**`--output` flag help text stale** (CFG-04 partial)
- File: `internal/cmd/root.go` line 113
- Current: `"output mode: stdout | sse | grpc | nats | sqs"`
- Should be: `"output mode: stdout | sse | grpc | nats | sqs | kafka | pubsub | rabbitmq"`
- Impact: `kaptanto --help` shows incomplete valid modes list. The runtime switch is correct and handles all 8 modes.
- Severity: Low — cosmetic discoverability issue; no functional impact
